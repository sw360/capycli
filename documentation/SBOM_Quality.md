# SBOM Quality

Yes, we all want high quality SBOMs.
But what is this high quality?

SBOMs generated by CaPyCLI are not intended to be shipped to customers
as offical CRA SBOMs.

CaPyCLI creates SBOMs that are intended for license compliance. They should be used
as input for creating components and projects on SW360.

## Definitions

### NTIA Framing Software Component Transparency

* **Author**: The Author reflects the source of the metadata, which could come from the creator of
the software being described in the SBOM, the upstream Component Supplier, or some third-party
analysis tool. Note that this is not the Author of the software itself, just the source of the
descriptive data.
* **Concluded License**: Frequently multiple licenses may be found in a Component that have
different constraints. After resolving the conditions an overall license for the Component
can be declared by the SBOM Supplier.
* **Included Component**: An Included Component is any Component that is in the distributed
software (e.g., masked layers within a container of an image).
* **Originating Supplier**: If the Component identified in the SBOM originated from a different
person or organization than identified as Component Supplier, the Originating Supplier defines
where or whom the Component originally came from. In some cases, a Component may be created
and originally distributed by a different third party than the Supplier of the Component.  
For example, the SBOM identifies the Component as glibc and the Component Supplier as Red Hat,
but the Free Software Foundation is the Originating Supplier.
* **Supplier**: The Supplier refers to the originator or manufacturer of the software Component.

### CycloneDX

* **Authors**: The person(s) who created the BOM. Authors are common in BOMs created through manual
processes. BOMs created through automated means may have `@.manufacturer` instead.
* **Component/Supplier**: The organization that supplied the component. The supplier may often be
the manufacturer, but may also be a distributor or repackager.
* **Component/Manufacturer**: The organization that created the component. Manufacturer is common
in components created through automated processes. Components created through manual means may have
`@.authors` instead.
* **Component/Authors**: The person(s) who created the component. Authors are common in components
created through manual processes. Components created through automated means may have
`@.manufacturer` instead.
* **Component/Publisher**: The person(s) or organization(s) that published the component
* **Component/Copyright**: A copyright notice informing users of the underlying claims to copyright
ownership in a published work.
* **Manufacturer**: The organization that created the BOM. Manufacturer is common in BOMs created
through automated processes. BOMs created through manual means may have `@.authors` instead.
* **Metadata/Supplier**: The organization that supplied the component that the BOM describes. The supplier
may often be the manufacturer, but may also be a distributor or repackager.

## Example: SBOM for CaPyCLI

The SBOM for CaPyCLI can be created with the command

```shell
capycli getdependencies python -i ./poetry.lock -o ./SBOM/sbom_no_dev.cyclonedx.json --search-meta-data
```

For CaPyCLI 2.10.1 this SBOM contains 45 components.  
Development dependencies like coverage, responses, pytest, isort, mypy, etc. are excluded by default.

When we analyze this SBOM with `sbomqs`, we don't get a perfect score:

### sbomqs Basic Check (NTIA)

```shell
sbomqs compliance --ntia ./SBOM/sbom_no_dev.cyclonedx.json
```

Result

```shell
NTIA Report
Compliance score by Interlynk Score:8.4 RequiredScore:6.7 OptionalScore:10.0 for ./SBOM/sbom_no_dev.cyclonedx.json
```

Issues for CaPyCLI 2.10.1

* primary component not declared
* `Package Supplier` missing for all components
  Well, Python does not know "suppliers". We can substitute it with authors.

Fixes in CaPyCLI 2.11

* Add primary component CaPyCLI to metadata
* Add CaPyCLI in the dependencies section
=> Compliance score by Interlynk Score:10.0 RequiredScore:10.0 OptionalScore:10.0 for .\sbom_for_qs.json

### sbomqs Framing Software Component Transparency (v3) Check

```shell
sbomqs compliance --fsct ./SBOM/sbom_no_dev.cyclonedx.json --color
```

Result

```shell
Framing Software Component Transparency (v3)
Compliance score by Interlynk Score:7.4 for ./SBOM/sbom_no_dev.cyclonedx.json
```

Issues for CaPyCLI 2.10.1

* optional: SBOM type not declared
* SBOM Relationship but dependency completeness missing for primary component
* Component checksum not declared
* Component copyright missing => no automatic way to determine for Python components!

Fixes in CaPyCLI 2.11

* add SBOM type, always `build`
* Add composition
  
  ```code
  "compositions": [
    {
      "aggregate": "unknown"
    }
  ],
  ```

* Add checksums for external references
* Add checksum of wheel file as component hash
* Add copyright text "N/A" as default for all components.

### sbomqs BSI Check (TR-03183-2, v1.1)

```shell
sbomqs compliance --bsi-v1 ./SBOM/sbom_no_dev.cyclonedx.json --color
```

Result:

```shell
BSI TR-03183-2 v1.1 Compliance Report
Compliance score by Interlynk Score:7.3 RequiredScore:8.0 AdditionalScore:5.1 for .\SBOM\sbom_no_dev.cyclonedx.json
```

Issues

* creator of SBOM is missing
  * author name is not sufficient, there must be an author email!
  * Adding a manufacturer with a url does not help.
  => Add a supplier with a url.
* component creator is missing
* source code URI is missing
* URI of deployable form is shown (`file:///urllib3-2.6.2-py3-none-any.whl`) but reported missing
* hash value of source code

Fixes:

=> **No fixes!** If someone wants a BSI compliant SBOM, then all missing data has to be searched
and added manually. The BSI guideline is only a recommendation, not a requirement.

## Why is the SBOM not signed

This is complex and not needed for the CaPyCLI use case.  
CaPyCLI uploads the SBOM to SW360, no signature check is needed or done.
